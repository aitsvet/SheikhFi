# AGENTS.md — метод этого репозитория

Кросс-инструментальный runbook. SheikhFi — референс-реализация web3-приложения
вида «бизнес-процесс на смарт-контракте + консольный webapp + testnet-демо»,
где предметные требования взяты из внешнего корпуса стандартов (здесь —
AAOIFI, исламские финансы). Чтобы построить **другое** приложение с другим
бизнес-процессом, меняется предметная часть (§«Как повторить»), каркас и метод
переносятся как есть.

## Рабочий цикл

Выведен из реальных сессий постройки и аудита этого репозитория; порядок —
принципиален.

1. **Тулбокс прежде всего.** Никаких установок на хост: каждый запуск —
   `docker compose run --rm node '<cmd>'`; версия node запинена образом.
   Полная проверка — `scripts/verify.sh` (тесты + coverage + lint + build +
   контейнерный e2e, с восстановлением активного деплоя).
2. **Аудит.** Прогнать всё, перечислить находки **от мелкого к крупному**:
   баги → UI-ложь → масштабирование → экономика/дизайн. Экономические
   претензии сверять с первоисточником стандартов (grep по корпусу,
   дословные цитаты), не с пересказами.
3. **Трассировка требований.** Каждый механизм получает строку в STANDARDS.md:
   пункт стандарта с цитатой, статус ✅/⚠️/❌. Каждый ❌ обязан получить раздел
   в PLAN.md — «сводка разрывов» и есть бэклог.
4. **PLAN.md как спецификация для слабого исполнителя.** Точные сигнатуры,
   инварианты, имена ревертов, требования к тестам, критерии приёмки — и
   блок правил (тулбокс; ABI-совместимость webapp через feature-detect; поля
   в структуры только в конец; коммиты одним словом по компонентам; push
   делает только владелец). Выполненный раздел схлопывается до записи с
   указателями — README описывает механику, STANDARDS — соответствие, тесты —
   поведение; спецификация не живёт дольше своего исполнения.
5. **Реализация волнами.** Раздел плана → тесты (сначала для экономики) →
   полный прогон → один однословный коммит. Отклонение от спецификации
   фиксируется в плане с обоснованием (пример: жадное списание убытка циклом
   вместо ленивого аккумулятора — ради точной пропорциональности).
6. **Тестирование требований — двухслойное.** Юнит-тест на каждый пункт
   (имена тестов входят в трассировку) плюс инвариантный сьют: seeded PRNG,
   ограниченные случайные операции всеми ролями, инварианты проверяются после
   каждого шага, легальные реверты — часть прогулки, panic — провал. Несущая
   функция цитирует свой пункт в NatSpec — `grep AAOIFI contracts/` даёт
   обратную трассировку из кода.
7. **Webapp против нескольких ABI.** Живой деплой не переезжает вместе с
   кодом: webapp определяет возможности по ABI (наличие функции, число
   inputs), поля структур читает по именам, не по индексам; без кошелька
   строит read-only контракт на публичном RPC (это же делает e2e осмысленным
   и даёт режим «регулятора»).
8. **E2E в compose.** Профиль `e2e`: цепочка → деплой → прод-сборка webapp →
   playwright. Ловит то, что юнит-тесты не видят (пример: vite preview
   блокировал Host compose-сервиса — нашлось только сквозным прогоном).
9. **Деплой.** Свежий кошелёк генерируется в `.env` (gitignored); газ — с
   программного крана (CDP — единственный без mainnet-gate); после деплоя —
   пер-чейн снапшот + onboarding участников скриптом; README-таблица деплоя
   обновляется; push (и, через CI, публикация демо) — за владельцем репо.

## Чек-лист аудита web3-DApp

Найденное здесь — типовое; проверять в первую очередь:

- **Экономика полного цикла:** возвращается ли принципал; отличается ли он от
  прибыли (fee с оборота — красный флаг); моделируются ли убытки; есть ли
  выход участника и по какой оценке (номинал — красный флаг).
- **Голосование:** вес голоса живой или замороженный; есть ли дедлайн и
  отмена; эмитится ли событие на голос; деление на ноль при пустом пуле.
- **Газ:** циклы в горячих путях (O(n²) голосование), рост стоимости с числом
  участников; оптимизатор включён?
- **Ключи и роли:** сменяемость owner (двухшаговая); что ломается при потере
  ключа; разделены ли операционные и надзорные роли.
- **Webapp:** захардкоженные значения вместо чтения контракта; полный рескан
  логов при каждом открытии; пустые `catch`; deprecated wallet-API.
- **Инфраструктура:** CI гоняет ли тесты перед публикацией; README дрейфует
  ли от автоматизации; секреты вне git; один перезаписываемый deployment-файл
  (красный флаг — нужны пер-чейн снапшоты).

## Как повторить с другим бизнес-процессом

1. **Корпус требований.** Замените AAOIFI на ваш источник (регуляторика,
   отраслевой стандарт, внутренние правила). Выпишите пункты с дословными
   цитатами в STANDARDS.md до того, как писать контракт: таблица «требование →
   пункт → (пока пусто) реализация → тесты» — это ТЗ.
2. **Контракт.** Смоделируйте роли и денежные потоки процесса; перенесите
   готовые механики по потребности: pull-payments (`withdrawable` + CEI),
   ленивый аккумулятор выплат (`cumulativePerShare`/checkpoint), замороженные
   веса голосов, транши с резервированием, залог под вердикт арбитра,
   ERC-20-представление доли, `_pull`/`_pay` для нативной/токенной
   деноминации.
3. **Тесты.** Скопируйте структуру: фикстуры-матрёшки по стадиям процесса +
   инвариантный сьют, где I-список выводится из вашей таблицы требований
   (платёжеспособность и «книги = токен» — почти универсальны).
4. **Webapp.** Оставьте каркас (`state.jsx` c `run`/`payIn`/feature-detect,
   hooks, deployments.js, read-only fallback), замените экраны и роли desk-ов
   под ваши.
5. **Инфраструктура — без изменений:** compose-тулбокс, e2e-профиль, CI,
   deploy.js, scripts/*. В `networks.js` — ваши сети.
6. **Ведение работ:** PLAN.md с правилами и волнами; аудит по чек-листу выше
   после каждой крупной волны; трассировка — единственный источник статуса
   соответствия.
