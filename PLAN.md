# PLAN — расширения SheikhFi (v3)

Спецификации для исполнителя. Каждый раздел самодостаточен: точные сигнатуры,
инварианты, требования к тестам и критерии приёмки. Выполнять по одному
разделу за раз, в указанном порядке; после каждого раздела — полный прогон.

Базовая версия — «экономика v2» (возврат тела, списание убытков, выход,
жизненный цикл предложения, двухшаговый ownership): реализована, покрыта
тестами и описана в README.md; трассировка на стандарты — в STANDARDS.md.
Ссылки вида «SS 12 3/1/5/4» — пункты стандартов AAOIFI, цитаты в STANDARDS.md.

## Правила для исполнителя (читать первым)

- Все команды — через compose-тулбокс, НЕ на хосте:
  `docker compose run --rm node '<команда>'`.
- Полный прогон = `npm ci && npx hardhat test` в корне; `cd webapp && npm ci
  && npm run lint && npm run build`; сквозной
  `docker compose --profile e2e up --abort-on-container-exit e2e`.
- v3 — новый деплой с новым ABI: менять сигнатуры МОЖНО, но webapp обязан
  работать с любым из деплоев в `webapp/src/abi/deployments/` через
  feature-detect по ABI (наличие функции; число inputs у фрагмента), как уже
  сделано для `hasEconomyV2`.
- В структуры `Proposal`/`Investor`/`Manager` новые поля добавлять ТОЛЬКО в
  конец; webapp читает поля по именам.
- Коммиты: одно слово, по компонентам, без trailer-ов. Push не выполнять —
  его делает пользователь (демо на gh-pages пересобирается CI при push в main).

## Волна v4 — верификация (следующий запуск)

Мотивация внешняя: доклад (`tahawwut/papers/TALK.md`) и статья
`cryptosarf/papers/CONSTRUCT.md` заявляют «формальные методы». Аудит
15.07.2026 (`tahawwut/papers/CLAIMS.md`) показал: единственный формальный
артефакт — TLA+-скетч, который (а) лежит в **приватном** `literature`,
(б) **никогда не прогонялся через TLC** (это признаёт его собственный
README), (в) описывает **модель** контракта, а не контракт.

**SheikhFi здесь первый и главный: он опубликован, а cryptosarf — нет.**
Тезис «правило шариата → код → машинная проверка» надо показывать на том
репозитории, который открывается по QR-коду со слайда. Сегодня в SheikhFi
верификации нет вообще.

Замеры (воспроизводимы):

### 1. Выбор инструмента верификации — сравнение

Требование: доказывать свойства **самого контракта**, в CI, бесплатно, без
новой экспертизы на месяцы. Кандидаты (обзор SOTA, июль 2026):

| Инструмент | Что проверяет | Что доказывает | Труд | Годен здесь |
| --- | --- | --- | --- | --- |
| **TLA+ / TLC** | **модель**, написанную руками | инварианты + темпоральные свойства (liveness) на ограниченной конфигурации, исчерпывающе | высокий | ❌ разрыв абстракции; у SheikhFi нет многосторонней темпоральной логики |
| **Quint** (Informal Systems) | то же, что TLA+ | то же; синтаксис приятнее, компилируется в Apalache/TLC | средний | ❌ тот же разрыв абстракции |
| **Halmos** (a16z) | **реальный контракт** (символьное исполнение) | утверждение верно **для всех входов** в ограниченном диапазоне; иначе — контрпример | **низкий** | ✅ **выбран** |
| **SMTChecker** (внутри solc) | **реальный исходник**, на компиляции | assert/переполнения/достижимость; движок CHC выводит индуктивные инварианты | **почти нулевой** | ✅ **выбран как базовый уровень** |
| **Kontrol** (Runtime Verification, KEVM) | реальный байткод против полной семантики K | сильнее всех; единственный с **loop invariants** → неограниченные циклы | высокий (нужен K) | ⚠️ избыточен для 599 строк |
| **Certora Prover** (CVL) | реальный байткод против спеки на CVL | сильные SMT-доказательства, индустриальный стандарт | высокий + **коммерческая лицензия** | ❌ цена |
| **hevm symbolic** | реальный байткод | как Halmos; плюс проверка эквивалентности двух байткодов | средний | ⚠️ ниша |
| **Echidna / Medusa / Foundry invariants** | реальный байткод | **ничего не доказывают** — ищут контрпример | низкий | ✅ но как фаззинг, не верификация |
| **Mythril** | байткод | поиск багов, не доказательство | низкий | ⚠️ дополняет |
| **Coq / Isabelle + extraction** | модель с извлечением кода | максимум строгости | месяцы | ❌ |

**Решение: Halmos (заголовок) + SMTChecker (базовый уровень).** Обоснование:

- **Нет разрыва абстракции — главный аргумент против TLA+.** TLA+ доказывает
  свойства модели, которую написали руками; ничто не связывает `.tla` с
  `.sol`, и расхождение никто не заметит. Halmos и SMTChecker доказывают
  свойства **того самого контракта, который задеплоен**. Для доклада, чей
  тезис — «код цитирует норму, которую исполняет», проверять надо **код**, а
  не рисунок кода. Это не компромисс: применительно к SheikhFi это
  **строго сильнее** TLA+.
- **Переиспользует уже написанное.** Инварианты I1–I6 из шапки
  `Invariants.test.js` — с выверенными ссылками на AAOIFI — становятся
  `check_*`-тестами Halmos почти дословно. Фаззер ищет контрпример, Halmos
  доказывает, что его нет ни при каком входе. Одни и те же формулировки.
- **Ограничение Halmos здесь не бьёт.** Слабость Halmos — неограниченные
  циклы и массивы (он разворачивает до заданной границы). В SheikhFi циклов
  два, и оба ограничены длиной массива; выплаты после аудита переведены на
  аккумулятор (`cumulativePerShare`) и pull-модель, то есть O(1).
  **Тот самый рефакторинг, который чинил газ, сделал контракт
  проверяемым** — это и есть сюжет для слайда 3.
- **SMTChecker бесплатен буквально**: движок CHC включается настройкой solc,
  новых инструментов в репозиторий не добавляется.
- TLA+ остаётся у CryptoSarf, где он оправдан (окна исполнения, терминальные
  статусы, двусторонний ва'ад — темпоральная логика, которой в SheikhFi
  нет), но обязан быть прогнан через TLC —
  `cryptosarf/tasks/045-verification-credibility.md`.

### 2. Halmos: что доказываем

- `foundry.toml` + `test/verify/Verify.t.sol` — Halmos работает поверх
  Foundry-тестов (Hardhat остаётся для юнит-тестов, деплоя и webapp).
- Перенести I1–I6 из шапки `Invariants.test.js` дословно, как `check_*` с
  символьными аргументами. Приоритет — свойства, которые фаззер способен
  только не опровергнуть:
  - **I2 Book equality** — `totalFunds == Σ fundsInvested == totalSupply()`
    для любых вкладов (SS 17 3/6). Арифметическое свойство: идеально для
    символьного доказательства.
  - **I6 Loss pro-rata** — списание сохраняет относительные доли при любой
    сумме убытка (SS 12 3/1/5/4); ±1 wei усечения — доказать границу, а не
    проверить на одном примере.
  - **I3 Accrual monotone** — `cumulativePerShare` не убывает ни при каком
    входе (SS 12 3/1/5/7).
  - **I1 Solvency** — баланс покрывает обязательства (pull-выплаты всегда
    исполнимы).
- Каждый `check_*` несёт `@custom:shariah AAOIFI-SS12-3/1/5/4` — тот же тег,
  что в NatSpec контракта (§3).
- Контейнером: `ghcr.io/a16z/halmos`; цель `make verify`. Границы
  (`--loop`, `--array-lengths`) — явно в конфиге и **названы в README**:
  доказательство ограниченное, и это надо писать прямо.

### 3. SMTChecker: базовый уровень даром

- В `hardhat.config.js`: `settings.modelChecker` — `engine: "chc"`,
  targets `assert`/`overflow`, таймаут. Прогон в CI отдельной целью
  (компиляция с CHC медленная — не в основной сборке).
- Что даёт: solc сам доказывает недостижимость `assert` и отсутствие
  переполнений индуктивно, **без границы** (в отличие от Halmos). Там, где
  CHC отвечает «proved», это сильнее Halmos; где «unknown» — работает Halmos.
  Честно показать оба результата.

### 4. Фаззер — чтобы слово было заслужено

Сегодняшний `test/Invariants.test.js` честно назван в шапке «the
hardhat-flavoured analogue of a Foundry invariant suite» — и это точное
самоописание: он **иллюстрирует** инварианты, а не ищет контрпримеры.

- **Один seed, одна траектория** (`mulberry32(20260711)`, 200 шагов) — это
  зафиксированный навсегда прогон.
- **Ревёрты проглатываются**: `catch (e)` пропускает всё, кроме `panic`.
  Значит, неизвестно, сколько из 200 шагов вообще что-то сделали. Ровно эта
  болезнь измерена в CryptoSarf: `exercise` — 494 вызова, **494 ревёрта**,
  инвариант «зелёный», потому что ничего не произошло. Здесь она не измерена
  вообще.
- **Нет усадки** — при падении на шаге 173 разбираться руками.

Делаем Foundry invariant suite (та же `foundry.toml`, что для Halmos):
хендлер по ролям, `bound()` на суммы, ghost-переменные
(`ghost_distributed`, `ghost_writtenOff`, `ghost_slashed`);
`runs = 256, depth = 64, fail_on_revert = true`. **Критерий приёмки —
таблица вызовов, а не зелёная галка**: у каждого изменяющего состояние
селектора `Reverts = 0`, плюс `invariant_campaignReachedTerminalStates()`,
чтобы возврат к пустоте падал в CI. `test/Invariants.test.js` остаётся
быстрым регрессом.

### 5. Трассировка: превратить STANDARDS.md в тест

`STANDARDS.md` — самый сильный артефакт репозитория. Не хватает только
машинной проверки, без которой слово «**автоматом**» не заслужено.

- Стабильный id пункта (`AAOIFI-SS12-3/1/5/3`), тег `@custom:shariah` в
  NatSpec (штатный механизм расширения Solidity) и в `check_*`/тестах.
- `scripts/check-traceability.mjs` в CI — красный при: цитируемый тест не
  существует/не проходит; символ отсутствует в контракте; тег ссылается на
  несуществующий пункт; строка ✅ без проходящего теста.
- Генерируемая карта «пункт → код → тест → доказано/протестировано».

**Это номер для доклада:** закомментировать проверку — CI краснеет и
называет пункт AAOIFI, который только что нарушен.

### 6. Числа

`102 passing` против «59 тестов» на слайде. Правки доклада —
`tahawwut/papers/CLAIMS.md`.

### Приёмка волны

- `make verify` — Halmos доказывает I1, I2, I3, I6; границы названы в
  README; счётчик доказанных/недоказанных честный.
- SMTChecker CHC прогнан; результаты (proved/unknown) записаны.
- `forge test --match-path "test/invariant/*" -vvv`: `fail_on_revert = true`,
  `runs = 256, depth = 64`, у всех селекторов `Reverts = 0`.
- `npx hardhat test` — по-прежнему 102+ passing.
- `check-traceability` красный на подсаженной регрессии (проверить реально).
- В репозитории и докладе слово «доказано» относится только к тому, что
  доказал Halmos/CHC; «проверено фаззером» — к остальному.
